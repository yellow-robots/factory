# Yellow Robots Dev Factory — Understanding

## What This Is

`yellow-robots/factory` is **infrastructure** — the autonomous build machinery that takes a **Ready** GitHub ticket to a **reviewed PR**, across any YR product repo. It is deliberately separate from the products it builds (`yellow-robots` = robot artifacts; `website`). A product repo declares *how* to build itself via a `.yr/factory.toml` manifest; the factory provides *the pipeline*.

## The Two Human Gates

The operating model has exactly **two human intervention points** — everything between them is autonomous:

1. **Promote to Ready** — a human decides the task meets the Definition of Ready and sets `Status = Ready` on the GitHub Project.
2. **Merge the PR** — a human reviews and merges the PR. (RFC 0007 accepted this as autonomous-merge-ready but it's not yet implemented.)

Everything else — implement, test, check, review, PR creation — runs unattended.

## The Pipeline (RFC 0002 — `dev-runner.sh`)

Each stage is a **separate cold `claude -p` process** — independence by construction (builder != verifier):

1. **DoR gate** — issue must be Open, in the project, Status=Ready, Type=Task, non-empty acceptance criteria. No LLM call on refusal.
2. **Claim** — Status -> In Progress (single-flight lock).
3. **Worktree** — fresh `git worktree` off `origin/main` of the target repo.
4. **Implement** — cold Claude process, `--permission-mode bypassPermissions` (confinement = environment, not prompt). Model: Sonnet default, Opus via `model: opus` in body.
5. **Test (independent)** — separate cold process, tests derived from acceptance criteria (the spec), NOT the implementation. **Boundary guard**: if tester changes anything outside `tests/`, the run is Blocked with the diff preserved for diagnosis. Build artifacts (`__pycache__/`, `*.pyc`) excluded from violation check.
6. **Check gate** — runner (not LLM) runs `check_cmd` from `.yr/factory.toml`. One repair attempt on **code** failure. **Environment** failure (exit 126/127) blocks without repair (no papering over broken toolchains).
7. **Review (independent)** — cold process emits `VERDICT: APPROVE` or `VERDICT: REQUEST_CHANGES`. Fail-closed: anything but exact `VERDICT: APPROVE` on the last line blocks.
8. **PR** — commit, push `task/<id>-<slug>`, open PR, Status -> In Review, post review verdict.

## Task State Model (RFC 0003)

All state lives on **native GitHub primitives**, not labels:

- **Type** = Issue Type (`Task` / `Bug` / `Feature`) — runner builds Tasks only.
- **Hierarchy** = native sub-issues (`gh issue create --parent`).
- **Status** = Projects single-select field: `Backlog -> Ready -> In Progress -> In Review -> Done`.
- **Reason** = separate Projects field for off-track work: `Needs-info`, `Blocked`.
- **Done** = native close (Projects auto-sets Status=Done on close).

## Dispatch (RFC 0004)

- **n8n** polls the board every 5 min via GraphQL, finds `Status=Ready & OPEN` items.
- POSTs each issue **with its explicit repo** to `dispatch.py` (host HTTP endpoint, bearer auth).
- `dispatch.py` fires `dev-runner.sh` under a `flock` (single-flight) detached.
- **Fail-closed dispatch** (RFC 0006): no default repo — a request without `owner/name` is refused and logged.
- Polling (not webhooks) is deliberate — self-healing, no missed events.

## Multi-repo (RFC 0006)

- One shared board ("Yellow Robots -- Dev") spans all product repos.
- Each item carries its native repository; n8n reads `repository.nameWithOwner`.
- Per-repo config lives in `.yr/factory.toml`: `check_cmd`, `model`, `base_ref`.
- Precedence: explicit env > manifest > built-in default.
- The factory discovers its workspace relative to itself (`YR_WORKSPACE`, default `factory/../..`), resolves target repos as `$YR_WORKSPACE/<name>`.

## Upper Pipeline (RFC 0005)

The path from product idea to buildable task, above `Ready`:

| Stage | Produces | Home | Gate |
|---|---|---|---|
| 1. Product intent -> spec | product spec (WHAT/WHY, EARS criteria, no tech) | Obsidian | spec ready |
| 2. Feature RFC | design handover | Obsidian | approve RFC |
| 3. Architectural assessment | technical RFC + per-task slices | epic GitHub Issue | review technical RFC |
| 4. Task decomposition | DoR task-Issues | GitHub Issues | promote to Ready |
| 5. Build -> PR | code | lower pipeline | merge |

Key conventions:
- **Self-contained handoff**: each stage gives the next a complete artifact; downstream never reaches back.
- **Traceability by citation**: every artifact cites its source (drift becomes detectable).
- **One-way airlock** (Obsidian <-> GitHub): crossed exactly once at feature-RFC -> technical-RFC by the "amphibian agent." Above: Obsidian. Below: GitHub, repo-local.
- **Gates review the plan, not the output**: approve outline before stage commits.

## Key Invariants

- **Builder != verifier**: implementer, tester, and reviewer are independent cold processes. Structural enforcement, not prompt-level.
- **Confinement = environment, not intent**: fresh worktree + scoped creds + deterministic gates. That's why `bypassPermissions` is safe.
- **Native primitives over sidecars**: Issue Types, Projects fields, sub-issues, native close -> Done. Not labels.
- **Deterministic gates dispose**: LLM proposes; machine-checked gate disposes. Nothing reaches main without passing a gate.
- **Repo-agnostic**: factory holds no product knowledge; product holds no copy of the factory.
- **Builds from git refs, never mutable working tree**: code and `.yr/factory.toml` read from `origin/main`.
- **One task = one PR**: bigger = split into sub-issues.
- **Docs consolidated, not accreted**: update/merge/trim canonical doc; don't pile new ones.

## Autonomous Merge (RFC 0007 — Accepted, Not Yet Implemented)

Replace human merge gate with: deterministic gates + one independent reviewer (Opus model) approves -> green build self-merges under a merge mutex + skew re-check. `main` != production (merge != ship). Future: multi-provider review panel, risk-tiering, post-merge safety net.

## Documentation Conventions (RFC 0009)

The Obsidian vault is the "design brain." Five doc types: `rfc`, `spec`, `research`, `note`, `runbook`. Lifecycle: `draft -> active -> superseded`. Filename is identity. `validate.py` enforces at the airlock; a flag-only linter surfaces vault drift.
